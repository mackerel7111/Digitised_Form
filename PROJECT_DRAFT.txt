DigitiseForm - Project Draft And Reference Notes
================================================

Purpose
-------

Build a small web application that:

1. Accepts a PDF form upload and creates a digital, fillable version.
2. Lets a user fill in that digital form and save the entry.
3. Lets the user download the original PDF with their entered values overlaid in the right places.

The main idea is not to recreate the exact look of the PDF in HTML. The app should create a practical digital form while storing enough mapping data to write the values back onto the original PDF later.


Recommended Architecture
------------------------

Use a hybrid pipeline:

PDF upload
  -> draft field extraction
  -> user review in template builder
  -> publish template
  -> fill digital form
  -> save submission
  -> overlay values onto original PDF

The parser only creates a first draft. The reviewed/published template is the source of truth.

This is important because VLM/OCR output may be imperfect, and PDF overlay needs reliable coordinates.

Why Vue + FastAPI
-----------------

Vue is comfortable for the interactive UI:

- PDF preview
- draggable boxes
- field inspector
- dynamic form rendering

FastAPI is a good fit for document processing:

- upload handling
- PDF page dimension extraction
- PDF overlay export
- parser integration
- SQLite persistence

PyMuPDF is useful because one library can:

- read PDF page sizes
- render pages to images if needed
- extract native text from text-based PDFs
- write overlay text back onto the PDF


Main User Interaction
---------------------

There are two major modes.

1. Template Builder

This is for the person adding a new form.

Flow:

- Upload PDF.
- App shows PDF preview.
- App creates a draft template if a parser is available.
- User reviews detected fields.
- User can rename fields, change field type, move boxes, resize boxes, delete false positives, and add missing fields.
- User publishes the template.

2. Form Filler

This is for the person filling a known form.

Flow:

- Pick a published form from the list.
- Fill in normal web inputs.
- Save entry.
- Download the filled PDF.


What The VLM Does
-----------------

The VLM should only be used to bootstrap the template.

Good VLM responsibility:

- read labels
- understand rough layout
- suggest fields
- suggest field types
- maybe suggest rough bounding boxes

Bad VLM responsibility:

- being the permanent source of truth
- writing directly to the final PDF
- replacing the review step

Reason:

PDF overlay is coordinate-sensitive. Even a smart parser can be slightly wrong. A human-reviewed template gives deterministic export behavior.


MinerU2.5 Pros And Cons
-----------------------

Pros:

- Built for document parsing.
- Better layout understanding than plain OCR.
- Can parse complex PDF structure.
- Makes the project more impressive if integrated cleanly.
- Useful as a draft generator for new forms.

Cons:

- May be hard for reviewers to run locally.
- May need GPU or heavy setup.
- Hugging Face/local model integration can take time.
- It may output document structure, Markdown, or layout blocks rather than exact fillable field mappings.
- Precise overlay coordinates still need to be stored and reviewed by the app.
- Cloud/API usage introduces external dependency, cost, privacy, and rate limit concerns.
- The linked model is AGPL-3.0, which is worth mentioning if discussing production use.

Conclusion:

Use MinerU2.5 as an optional parser backend, not as the whole system.


Non-VLM Extraction Approach
---------------------------

If the PDF has selectable text:

- Use PyMuPDF or pdfplumber to extract words and coordinates.
- Detect labels such as "Name", "Date", "Address".
- Detect nearby blank lines, boxes, and table cells.

If the PDF is scanned:

- Render each page to an image with PyMuPDF.
- OCR the image using Tesseract, PaddleOCR, or EasyOCR.
- Use OpenCV to detect lines, rectangles, and checkboxes.
- Associate nearby labels with detected empty regions.

This approach is more transparent and easier to test, but may require more heuristic tuning.

-------------------

Example template:

{
  "id": "sample-1",
  "name": "Sample Form 1",
  "pdfFileId": "sample-1.pdf",
  "status": "published",
  "pages": [
    {
      "page": 1,
      "width": 595,
      "height": 842
    }
  ],
  "fields": [
    {
      "id": "full_name",
      "label": "Full Name",
      "type": "text",
      "page": 1,
      "rect": {
        "x": 0.18,
        "y": 0.31,
        "w": 0.42,
        "h": 0.025
      },
      "style": {
        "fontSize": 10,
        "align": "left"
      }
    }
  ]
}

Field types to support first:

- text
- multiline
- date
- number
- checkbox

Later field types:

- radio group
- dropdown
- signature placeholder


Coordinate Mapping
------------------

The browser preview uses screen pixels and top-left origin.

PDF writing usually uses PDF points and bottom-left origin.

Store normalized coordinates:

x = box_left / rendered_page_width
y = box_top / rendered_page_height
w = box_width / rendered_page_width
h = box_height / rendered_page_height

During export:

pdf_x = rect.x * page_width
pdf_y = page_height - ((rect.y + rect.h) * page_height)

For text:

- Draw the submitted value at pdf_x, pdf_y.
- Use the field's saved font size.
- For multiline text, wrap text within the field width.

For checkbox:

- If true, draw an X, check mark, or filled square inside the rect.

API Sketch
----------

POST /api/forms/upload

- Upload original PDF.
- Return form id and page metadata.

POST /api/forms/{form_id}/draft-template

- Run optional parser.
- Return draft fields.

PUT /api/templates/{template_id}

- Save edited template.

POST /api/templates/{template_id}/publish

- Mark template as published.

GET /api/templates

- List published templates.

POST /api/templates/{template_id}/submissions

- Save filled values.

POST /api/submissions/{submission_id}/export

- Generate and return filled PDF.

Report Draft: Three Decisions
-----------------------------

Decision 1: Hybrid extraction instead of full automation

I treated AI/OCR extraction as a draft generator rather than the source of truth. Form field detection can be slightly wrong, especially on scanned or irregular PDFs, and small coordinate errors matter during PDF export. A human-reviewed template keeps the final overlay deterministic while still using automation to reduce setup time.

Decision 2: Store normalized coordinates

Field boxes are stored as page-relative normalized coordinates instead of screen pixels. This makes templates independent of browser zoom, monitor size, and PDF preview scale. During export, normalized values are converted into PDF points, including the top-left to bottom-left coordinate conversion needed for PDF writing.

Decision 3: Keep MinerU optional

MinerU2.5 is a strong candidate for document parsing, but requiring a local VLM could make the app harder to clone and run. I kept parser integration behind an interface so the core upload, template editing, form filling, and PDF overlay flow works without GPU setup. MinerU can be added as an optional draft extraction backend.
